Transfer an Employee and/or Update Employee Manager Usecases

Summary​

The HR use case for transferring an employee involves moving an employee from one position, department, or location to another within the organization. This process includes updating job titles, responsibilities, and reporting structures, as well as managing any necessary adjustments to compensation, benefits, and training.

Value Statement: Automating employee transfers enhances efficiency by streamlining the updating of job roles, compensation, and reporting structures. This ensures a smooth transition with minimal manual intervention, reducing errors and improving employee experience and operational continuity.

IBM has successfuly implemented this usecase in IBM AskHR. The current customer POC's are in progress.

danger

This documentation is currently under construction. Refer to the flight log documentation to stay up to date on the development efforts or check back soon

Prerequisites​

IBM Technologies​

  1. IBM watsonx Orchestrate

Supporting Technologies​

  1. Oracle HCM with GET access to "workers" API and ability to perform a PATCH functions

Oracle HCM Process Flow​

Each process flow requires following parameters:

  1. EmployeeID (Parsed from GET worker call - ID located in the "self" child URL, and returns multiple workers unless a query parameter is used)
  2. PeriodOfServiceId (Using the workersUniqID parameter above, property is directly returned by the GET worker workRelationships call, in the workRelationships child)
  3. Worker Assignment (Using the above 2 parameters, parsed from GET worker workRelationships call - ID located in the in the "assignments" child URL). These requirements accomplish "Transfer an Employee" with PATCH Update a Worker Assignment
  4. Worker Manager (Using the above 3 parameters returns the workerManagers child). These requirements accomplish "Update a Worker Manager" with PATCH Update a Worker Manager.

You must follow the chain of returns from the GET worker call, GET worker workRelationships call, and GET worker assignments call and retain the "uniqId" in the URL to pass to the required parameters for the PATCH calls (Update a Worker Assignment and Update a Worker Manager) API.

warning

This documentation assumes that for each worker there is a single assignment and a single manager

Transfer an Employee​

Refer to the Oracle HCM "Transfer a Worker" Documentation

Oracle HCM API PATCH Path

/hcmRestApi/resources/11.13.18.05/workers/{workersUniqID}/child/workRelationships/{PeriodOfServiceId}/child/assignments/{assignmentsUniqID}

API Process Flow​

Example PATCH input to transfer an employee's location:

{
"ActionCode": "TRANSFER",
"ReasonCode": "LOCCHANGE",
"LocationId":"300100073991592"
}

Update Employee Manager​

Similarly to update a manager, it requires the above children URLs and goes one child further down the path in order to update an employee's manager anf accomplish the "Update Employee Manager" usecase.

Refer to the Oracle HCM "Update a Manager" Documentation

Oracle HCM API PATCH Path:

/hcmRestApi/resources/11.13.18.05/workers/{workersUniqID}/child/workRelationships/{PeriodOfServiceId}/child/assignments/{assignmentsUniqID}/child/managers/{managersUniqID}

API Process Flow​

Example PATCH input to change an employee's manager:

{
"ActionCode": "MANAGER_CHANGE",
"ManagerAssignmentNumber": "E955160008191552",
"ManagerType": "LINE_MANAGER"
}

  3. The optional query parameters must be set per the yaml comments or left blank
  4. The input parameters must match the above example's format
danger
  1. This yaml cannot be imported to watsonx Orchestrate directly, as it does not include all watsonx Orchestrate required properties
  2. An intermediary application is required to parse and pass the uniqId for each parameter where required sourced from the URL in the GET call responses, as watsonx Orchestrate does not currently support complex output parsing in Skills

          in: query
          description: >
            This query parameter defines the where clause. The resource collection will be queried using the provided expressions. 
            The value of this query parameter is one or more expressions.

            You can use these queryable attributes to filter this collection resource using the `q` query parameter:
            
            - `DisplayName`: string; Person's display name.

            - `FullName`: string; Person's full name.

            type: string

              type: object
